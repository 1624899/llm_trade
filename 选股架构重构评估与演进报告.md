# 选股架构重构评估与演进报告

## 一、当前重构结论

将原来的 `ScreenerAgent` 重构为纯规则 `StockScreener`，并把它提升到整个选股工作流的最上层，是一次正确的架构收敛。

新的主流程应该明确为：

```text
技术预筛 StockScreener，纯规则
    ↓ 5-10 只候选
宏观分析 MacroAgent，LLM
    ↓
深度分析 FundamentalAgent + TechnicalAgent + NewsRiskAgent，LLM
    ↓
决策输出 DecisionAgent，LLM
```

这个结构的核心价值是：**让确定性规则处理全市场，让 LLM 只处理少量高价值候选股**。这样既降低幻觉风险，也降低调用成本和运行时间。

## 二、当前实现的优势

### 1. 初筛稳定性明显提升

重构前，Text2SQL 依赖 LLM 生成 SQL。一旦模型生成了错误字段、错误表名、不合理 `LIMIT`，或者非预期查询，流程第一步就可能失败。

重构后，`StockScreener` 使用本地 `daily_quotes` 和 `market_bars` 做确定性技术筛选，规则包括排除创业板、科创板、北交所、ST、退市整理、近 5 日停牌、近 5 日跌停、高波动、高量比、过度超买和低流动性标的。候选池质量更可控。

### 2. LLM 的职责更清晰

LLM 不再负责“从全市场里找股票”，而是负责它更擅长的事情：

- 解释候选股的基本面变化
- 复核技术形态和风险点
- 检查近期减持、监管、诉讼、重组失败、业绩预警等资讯雷点
- 结合宏观环境做相对排序
- 输出可读的决策报告

这比让 LLM 先生成 SQL、再解释结果更稳，也更容易测试。

### 3. 漏斗模型更符合实盘工作流

当前架构已经形成较合理的漏斗：

```text
全市场约 5000 只
    ↓
规则预筛 5-10 只
    ↓
宏观环境定调 1 次
    ↓
候选股深度分析 5-10 次
    ↓
最终推荐 1-3 只
```

这个漏斗比“让 Agent 在全市场中自由发挥”更接近真实投研流程。

## 三、当前主要不足

### 1. 筛选规则仍然硬编码

当前阈值写在 `screener_agent.py` 中，例如：

- 20 日年化波动率上限 80%
- 量比上限 5.0
- 总市值下限 30 亿
- MA5 高于 MA20 超过 15% 淘汰
- 换手率低于 0.5% 或高于 20% 淘汰

这些规则本身合理，但市场环境变化很快。牛市、震荡市、熊市应该使用不同的筛选阈值。如果全部硬编码，后续调参会很不方便。

### 2. 缺少淘汰原因统计

现在系统能给出入选股票，但不容易知道大量股票是被什么规则淘汰的。

例如一次预筛后，系统最好能输出类似统计：

```text
退市/ST 淘汰：32 只
近 5 日停牌或跌停淘汰：81 只
波动率过高淘汰：416 只
市值不足淘汰：590 只
量比过高淘汰：126 只
技术趋势不达标淘汰：2100 只
最终入选：10 只
```

这类统计对调参很重要。否则规则太严还是太松，只能靠感觉判断。

### 3. 宏观上下文已开始传导，但还需要继续强化

宏观 Agent 能判断市场环境。当前实现已经将 `macro_context` 传给 `FundamentalAgent`、`TechnicalAgent` 和 `DecisionAgent`，使市场风格能进入后续 prompt。

后续仍需要继续强化的是：让各 Agent 的输出格式更结构化，明确体现它们如何响应宏观基调。例如宏观判断是“风险偏好下降、偏防守”，那么后续 Agent 应该更关注：

- 高估值和高波动风险
- 业绩稳定性
- 成交拥挤度
- 技术破位后的止损条件

这些已经进入后续 prompt，但未来最好进一步沉淀为结构化字段，便于最终决策做硬约束。

### 4. 资讯防雷已补入，但数据源覆盖仍需增强

当前实现已经新增 `NewsRiskAgent`，专门检查候选股近期新闻中的减持、监管函、处罚、诉讼、重组失败、业绩预警等风险，并把结果传给最终决策。

仍需注意的是，免费新闻源覆盖不一定完整。后续可以继续增强：

- 公告类信息：减持、质押、业绩预告、重组终止
- 监管类信息：问询函、监管函、立案调查、行政处罚
- 舆情类信息：重大负面新闻、核心订单取消、行业政策利空

这些信息如果能结构化进入 `NewsRiskAgent`，防雷能力会更稳定。

### 5. Phase 3 串行执行，运行时间偏长

候选股之间的深度分析互不依赖，目前如果按顺序分析 10 只股票，每只股票调用基本面、技术面和资讯防雷 Agent，整体耗时会线性增长。

这部分适合并发执行。短期不一定要全量改成 `asyncio`，用 `ThreadPoolExecutor` 就可以明显缩短等待时间，改动也更小。

### 6. 持仓生命周期还不完整

当前系统偏向“推荐买入”，但完整流程还应该包括：

- 已推荐股票是否继续持有
- 是否触发减仓或清仓
- 是否达到止盈或止损条件
- 亏损案例是否进入反思和归因

否则系统容易变成只会不断推荐新票，而不是管理一个持续演化的观察仓。

## 四、演进优先级建议

### P0：规则配置化

建议新增或完善 `config/stock_picking.yaml`，把筛选阈值从代码中迁移出来。

建议先支持三个 profile：

```yaml
profiles:
  conservative:
    annual_vol20_max: 0.65
    volume_ratio_max: 4.0
    min_total_market_cap: 5000000000
    max_ret20: 25

  balanced:
    annual_vol20_max: 0.80
    volume_ratio_max: 5.0
    min_total_market_cap: 3000000000
    max_ret20: 35

  aggressive:
    annual_vol20_max: 1.00
    volume_ratio_max: 6.0
    min_total_market_cap: 2000000000
    max_ret20: 45
```

短期可以默认使用 `balanced`，后续再根据市场状态自动切换。

### P1：增加淘汰原因与筛选审计

`StockScreener` 不仅要返回入选候选，还应该记录：

- 每条规则淘汰了多少股票
- 每只未入选股票的主要淘汰原因
- 最终候选股的入选理由和关键指标

建议输出到日志，后续也可以落盘到 `outputs/screener_audit.json`。

这会让调参和排错容易很多。

### P2：宏观上下文结构化

`MacroAgent` 的输出已经传给下游 Agent：

- `FundamentalAgent.analyze(code, name, macro_context=None)`
- `TechnicalAgent.analyze(code, name, macro_context=None)`
- `DecisionAgent.synthesize_and_elect_winners(..., macro_context=None)`

下一步应把宏观字段进一步标准化，例如 `risk_appetite`、`liquidity_view`、`favorable_sectors`、`key_risks`，并要求下游 Agent 在输出中显式回应这些字段。

### P3：资讯防雷数据源增强

`NewsRiskAgent` 已经成为独立防雷关卡。下一步建议补充公告和监管信息源，优先级高于扩展更多泛新闻源。

建议重点接入或缓存：

- 股东减持、质押、解禁
- 问询函、监管函、处罚公告
- 业绩预告、业绩快报修正
- 重组终止、定增失败、重大诉讼

### P4：候选股深度分析并发化

Phase 3 可以先用线程池改造：

```text
候选股 A：FundamentalAgent + TechnicalAgent + NewsRiskAgent
候选股 B：FundamentalAgent + TechnicalAgent + NewsRiskAgent
候选股 C：FundamentalAgent + TechnicalAgent + NewsRiskAgent
...
```

短期建议限制并发数，例如 3-5，避免触发 LLM API 限流。

### P5：持仓诊断与卖出规则

在 `--post` 或 `--pick` 前增加观察仓诊断：

- 继续持有
- 减仓观察
- 清仓退出
- 触发反思

每条建议都要给出原因，例如技术破位、放量下跌、基本面利空、市场风格切换等。

## 五、对 ThemeAgent 的建议

板块主线和市场情绪在 A 股里很重要，但不建议一开始就新增完整 `ThemeAgent`，也不建议让“主线标的无条件进入深筛池”。

更稳的方式是先做轻量级 `ThemeScorer`：

- 使用新浪免费行业/概念板块数据
- 统计板块涨跌幅、上涨家数、成交额活跃度
- 给候选股增加主题热度加分
- 不绕过退市、停牌、跌停、高波动、流动性不足等硬风控

也就是说，主题可以加分，但不能越过底线风控。

## 六、推荐落地路线

建议按下面顺序推进：

1. `StockScreener` 阈值配置化
2. 增加筛选淘汰原因统计
3. 将 `macro_context` 标准化，并要求下游 Agent 显式回应宏观基调
4. 增强 `NewsRiskAgent` 的公告和监管数据源
5. Phase 3 使用线程池并发分析候选股
6. 增加观察仓诊断和止盈止损建议
7. 增加轻量级 `ThemeScorer`，作为评分加分项
8. 最后再考虑市场状态自动切换 profile

## 七、总体评价

当前重构方向是正确的：**把规则层和 LLM 层拆开，把筛选器放到 Agent 体系上层，把 LLM 用在少量候选股的深度分析上**。

接下来最重要的不是继续堆更多 Agent，而是让系统更可调、更可解释、更快、更闭环：

- 可调：规则配置化
- 可解释：记录淘汰原因
- 更快：候选股并发分析
- 更一致：宏观上下文结构化贯通
- 更谨慎：资讯防雷独立成关
- 更闭环：持仓诊断、止损、反思归因

做到这些之后，再扩展板块主线、市场状态识别和产业链推理，系统会稳得多。
