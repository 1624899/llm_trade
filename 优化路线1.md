我觉得 `go-stock` 最值得搬到你这个项目里的，不是选股规则本身，而是“把研究系统产品化”的那一层。

你现在的 `LLM-TRADE` 在核心闭环上更扎实：本地数据湖、多策略规则雷达、遮盖式走步回测、观察仓、交易仓、退出和亏损反思都已经写进 [README.md](D:/桌面/llm-trade/README.md:3)。`go-stock` 的强项则是桌面端体验、行情展示、预警、多模型接入、新闻/研报/财经日历这些面向日常使用的能力。它的 README 显示项目基于 Wails + NaiveUI，支持 A 股/港股/美股、市场/个股情绪分析、K 线技术分析、涨跌报警、本地数据保留，并接入 OpenAI、Ollama、LMStudio、DeepSeek 等模型平台。

可以优先借鉴这几块：

1. **做一个可视化工作台**
   你现在主要输出 `outputs/latest_report.md`、`latest_workflow_audit.json`、`latest_agent_trace.jsonl`。可以先不重写成 Wails 桌面端，先加一个轻量 Web UI：展示今日候选、观察仓、交易仓、Agent 结论、回测权重、亏损反思。  
   优先级最高，因为这会立刻降低每天使用成本。

2. **观察仓预警系统**
   `go-stock` 有涨跌报警推送。你的项目已经有 `Watchlist`、`ExitAgent`、`TradingAccount`，很适合加规则预警：
   - 跌破 MA20 / ATR 止损
   - 接近买点区间
   - 放量滞涨
   - 观察仓浮亏超过阈值
   - 持仓触发清仓/减仓信号  
   先输出到控制台和 `outputs/alerts.json`，再接企业微信、钉钉、邮件。

3. **多模型 Provider 抽象**
   `go-stock` 支持 OpenAI-compatible、Ollama、LMStudio、DeepSeek 等。你现在可以把 Agent 调用层抽象成统一 `LLMProvider`，让本地模型跑快速筛选，云模型跑深度复核。这样成本和稳定性都会好很多。

4. **自定义 Prompt 模板**
   `go-stock` 已有自定义 AI 分析提问模板和多轮对话能力。你这里可以把 `MacroAgent`、`FundamentalAgent`、`TechnicalAgent`、`DecisionAgent` 的 Prompt 模板外置到 `config/prompts/`，并给每个模板加版本号。  
   这对后续回测“Prompt 版本影响”也很有价值。

5. **市场资讯、公告、研报、财经日历面板**
   `go-stock` 更新日志里提到市场资讯 AI 总结、公司公告、个股研报、财经日历、重大事件时间轴。你项目已有 `NewsRiskAgent`，可以把这些作为结构化输入源，而不是只做搜索摘要。  
   推荐落地为：`market_events` 表 + `company_announcements` 表 + `research_notes` 摘要表。

6. **资金流向和行业热度**
   `go-stock` 有资金趋势、行业排名、热门股票/事件/话题等产品功能。你可以把它们接入 `MarketRegimeDetector` 和 `theme_scorer.py`，让“市场状态”和“主题加分”更接近真实盘面，而不是只靠涨跌停和指数。

7. **报告二次追问**
   `go-stock` 支持 AI 分析后继续对话。你这里可以基于 `latest_report.md`、`latest_workflow_audit.json`、`latest_agent_trace.jsonl` 做一个“问报告”功能，例如：
   ```bash
   python main.py --ask-report "为什么今天没有推荐新能源？"
   python main.py --ask-report "观察仓里哪只最该移除？"
   ```

8. **自动更新和打包**
   `go-stock` 有频繁 release 和自动更新能力。你的项目如果以后给自己长期使用，可以加：
   - `config_example.yaml`
   - 一键初始化命令
   - Windows `.bat` / PowerShell 启动脚本
   - 每日任务计划模板
   - 版本号和迁移脚本

我的建议路线是：

第一阶段先做 **Web 工作台 + 观察仓预警**。  
第二阶段做 **多模型 Provider + Prompt 模板版本化**。  
第三阶段再补 **公告/研报/财经日历/资金流向**。  
桌面端 Wails 可以最后考虑，因为现在真正缺的不是壳，而是把你已有的研究闭环变成每天好用的操作台。
